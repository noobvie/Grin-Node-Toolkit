  ── MINING CORE ─────────────────────────────────────────────────────────────

  BUG-01  Stratum protocol is BITCOIN, not GRIN.
          lib/stratum-server.js, lib/stratum-protocol.js
          Implements mining.subscribe / mining.submit / mining.set_difficulty.
          Grin uses login / getjobtemplate / submit / keepalive with
          {nonce, pow[42], edge_bits}. No real miner will connect.
          FIX: Port grin-mining-pool's stratum or reimplement against
          https://github.com/mimblewimble/grin-miner stratum spec.

  BUG-02  No block-finding pipeline exists.
          lib/block-monitor.js only polls network tip height. There is no
          code that watches the pool wallet for coinbase outputs and calls
          blockManager.creditBlock(). The `blocks` table is therefore
          NEVER populated → reward/payout/orphan chain has zero input.
          FIX: Add a watcher that listens for coinbase TX in the pool
          wallet, correlates to the block height, and calls creditBlock().

  BUG-03  PPLNS double-pay: shares have no "paid" marker.
          lib/rewards.js  getSharesForDistribution() (line 99)
          If two blocks land within 60 heights, overlapping shares are
          credited TWICE. There is no shares.paid_in_block_id column.
          FIX: Add column `shares.paid_in_block_id INTEGER NULL`. Set it
          inside the distribution transaction. Filter `WHERE paid_in_block_id
          IS NULL` in getSharesForDistribution.

  BUG-04  distributeRewards() is never called.
          Grep confirms zero callers. block-monitor confirms blocks via
          orphan-detector but never triggers distribution.
          FIX: In block-monitor.checkImmatureBlocks(), after
          orphanDetector.confirmBlock(), call rewardDistributor.distributeRewards(blockId).

  BUG-05  RewardDistributor.grinNode is undefined.
          lib/rewards.js line 24 references this.grinNode; the constructor
          never assigns it. Blockchain verification ALWAYS skipped.
          FIX: Constructor takes (config, grinNode). index.js must pass it.

  BUG-06  Stale-share check rejects every share.
          lib/shares.js line 23: Date.parse(shareHash) on a SHA-256 hex
          string returns NaN → shareAge is huge → every share is "stale".
          FIX: Pass a real timestamp into submitShare() — the share hash
          is not a date.

  BUG-07  currentJobId never increments + server.connections deprecated.
          lib/stratum-server.js line 20, 207, 227
          Only jobId 0 is ever "current"; everything else is rejected.
          server.connections was removed from Node ≥ 12.
          FIX: Track sockets in a Set; bump currentJobId on each new job.

  BUG-08  TCP packet fragmentation breaks stratum parsing.
          lib/stratum-server.js  data.toString().split('\n') with no
          per-socket buffer.
          FIX: Maintain `socket.recvBuf` and only parse complete lines.

  ── WALLET / MONEY FLOW ─────────────────────────────────────────────────────

  BUG-09  Withdrawal accounting never locks funds.
          lib/withdrawal-scheduler.js
          initiateWithdrawal() doesn't move balance → balance_locked.
          markFailed() then does balance += amount, balance_locked -= amount.
          Since locked was never incremented, the user's balance DOUBLES on
          every failure. Also miners can request more than their balance
          before any complete.
          FIX:
            - On withdrawal create: atomic UPDATE setting balance = balance-X,
              balance_locked = balance_locked+X (only if balance >= X).
            - On confirm: balance_locked -= X.
            - On failure: balance_locked -= X, balance += X.
            - Wrap all three in a sqlite transaction.
            - Add CHECK constraints: balance >= 0, balance_locked >= 0.

  BUG-10  canInitiateWithdrawal() defined but never called.
          MAX_PENDING_WITHDRAWALS / MAX_USER_PENDING limits don't fire.
          FIX: Call it at the top of initiateWithdrawal().

  BUG-11  Command injection in WalletTor.
          lib/wallet-tor.js line 20
          `grin-wallet -p ${dir} send -d '${addr}' -a ${amt}` then
          spawn('bash', ['-c', cmd]).
          FIX: spawn('grin-wallet', ['-p', dir, 'send', '-d', addr, '-a', String(amt)])
          with no shell.

  BUG-12  WalletTor regex matches .onion, not Grin Slatepack addresses.
          Withdrawals use grin1.../tgrin1... addresses routed via Tor —
          NOT raw .onion. isTorAddress("grin1abc...") returns false →
          every withdrawal gets scheduled for retry forever.
          FIX: Rename to isGrinAddress(); use /^t?grin1[a-z0-9]{55,62}$/.
          Drop the raw-TCP "tor probe" entirely — grin-wallet handles
          Tor routing internally via its own SOCKS config.

  BUG-13  WalletAPI uses Owner API v3 without ECDH.
          lib/wallet.js line 96 hits /v3/owner with plain Basic Auth.
          Per CLAUDE.md, v3 requires init_secure_api → ECDH → AES-GCM.
          FIX: Either implement v3 handshake (preferred long-term), or
          switch to Owner API v2 endpoints + methods for MVP.

  BUG-14  WalletAPI.sendTransaction calls non-existent method 'send'.
          Owner API has init_send_tx, not send.
          FIX: Use init_send_tx + finalize_tx flow.

  BUG-15  GrinNodeAPI Foreign endpoints unauthenticated.
          lib/grin-node.js line 107: secret only sent for Owner API.
          Per CLAUDE.md, Foreign API also needs Basic Auth with
          .foreign_api_secret. getTip / getHeader / getBlock all 401.
          FIX: Take both secrets in constructor; send the right one per
          endpoint. Or load each from its own file path at call time.

  BUG-16  Money in JS floats (REAL columns + Number).
          Cumulative rounding errors on credits/debits.
          FIX: Store nanoGRIN as INTEGER (1 GRIN = 1e9 nanoGRIN). Use
          BigInt in JS. Convert to GRIN nominal only at the display layer.

  BUG-17  Orphan reversal is wrong (and unsafe).
          lib/orphan-detector.js line 138: reverses block.reward / shares.length
          — equal split — but distribution was PPLNS weighted by difficulty.
          Reverses miner credits only, not pool fee. No guard against
          negative balance if miner already withdrew.
          FIX: Read the actual credited amounts from balance_log
          (or shares.paid_in_block_id × that share's payout column) and
          reverse those exact amounts. Reverse pool_fee_address credit too.
          Refuse to reverse if it would push balance < 0; flag for admin.

  ── AUTH / DATA INTEGRITY ───────────────────────────────────────────────────

  BUG-18  admin_audit_log schema mismatch — every INSERT throws.
          lib/db.js   columns: admin_id, action, target_type, target_id,
                       before_state, after_state, ip
          index.js    INSERTs:  user_id, action, resource, details
          pool-settings.js writes:  admin_id, action, target_type, target_id, details
          target_type is NOT NULL → every login/register/update audit row
          silently fails (caught and logged). Audit log is permanently empty.
          FIX: Pick one column shape. Recommended: rename to
          (admin_id, action, target_type, target_id, details JSON, ip TEXT,
          created_at INTEGER). Update all 6+ writers.

  BUG-19  Frontend auth guard always passes.
          public_html/js/auth.js line 119-123:
            function requireAuth() {
              if (!Auth.isLoggedIn()) { … }   // Promise → always truthy
            }
          Same bug in:
            public_html/login.html line 390 — hits /api/health (public, 200
              for everyone) → redirects every visitor to admin-dashboard.html
              before they even log in.
            back-end-pool/admin-panel/index.html line 124 — API.isLoggedIn()
              is a Promise; truthy; admin page never blocks unauth.
          FIX: All three: await the check. Use an authenticated endpoint
          like /api/admin/dashboard (returns 401 for unauth) — not /api/health.

  BUG-20  Spoofable X-Forwarded-For in admin allowlist.
          lib/ip-filter.js line 229; lib/rate-limiter.js line 214
          Anyone with `X-Forwarded-For: <allowed-ip>` bypasses both.
          FIX: In index.js do app.set('trust proxy', 1) and use req.ip
          everywhere. Strip the header-trust code in these two libs.

  BUG-21  JWT refresh tokens replayable.
          lib/auth.js refreshAccessToken doesn't invalidate the old refresh
          token. Stolen tokens valid for the full 7 days.
          FIX: Track refresh token jti in a refresh_tokens table; mark used
          jtis as revoked on refresh.

  BUG-22  No account lockout on failed login.
          Rate-limit is per-IP only. Attacker rotates IPs to brute-force.
          FIX: Add failed_login_attempts counter per username; lock after N.

  BUG-23  bcrypt rounds = 10 (too low for 2025).
          lib/auth.js line 237   bcrypt.genSalt(10)
          FIX: Use 12 minimum.

  ── ALERT MONITOR ───────────────────────────────────────────────────────────

  BUG-24  AlertMonitor reads fields/methods that don't exist.
          lib/alert-monitor.js
            line 118: status.sync_status === 'Synced'  (grin-node returns
                      {ok, height, total_difficulty, network, timestamp})
            line 155: this.walletTor.checkHealth()      (method doesn't exist)
            line 208: stratumServer.getStats() reads shares_accepted_1h,
                      connection_errors_1h (neither field exists)
            line 256: WHERE status='failed'              (column is 'tor_failed')
            line 256: datetime('now',...)                (column is unixepoch int)
          Net effect: critical wallet + node alerts fire EVERY check cycle.
          FIX: Align field/method names; standardize on unixepoch timestamps;
          add `withdrawals.updated_at` column or query by `created_at`.

  BUG-25  alert_thresholds stored as JSON string, read as object.
          lib/pool-settings.js line 67 stores '{...}' as text.
          lib/alert-monitor.js reads config.alert_thresholds?.X — undefined.
          FIX: Mark value_type='json' in pool-settings storage, or JSON.parse
          when reading.

  BUG-26  Alert delivery is stubbed in alert-monitor itself.
          alert-monitor.deliverAlert() just console.logs "Would send …".
          alert-delivery.js has real implementations but is never invoked.
          FIX: Pass alertDelivery into AlertMonitor constructor; call
          alertDelivery.send(alert) instead of the stubs.

  ── XSS / FRONTEND HARDENING ────────────────────────────────────────────────

  BUG-27  11 innerHTML = data.map() sites; only one file defines escHtml.
          See review for full list. Worker names, status reasons, settings
          values are stored-XSS sinks.
          FIX: Move escHtml() to /js/escape.js and use it in every map.
          Or switch to DOM construction (createElement + textContent).

  BUG-28  CSP allows 'unsafe-inline' for scripts.
          index.js line 47. Required because every HTML page uses inline
          onclick= and inline <script>. CSP is decorative.
          FIX: Move all inline handlers to addEventListener in separate
          .js files. Then drop 'unsafe-inline'. Add nonce-based CSP.

  BUG-29  SVG uploads accepted by AssetManager.
          lib/asset-manager.js line 14. SVG can contain <script> →
          stored XSS via admin-uploaded logo.
          FIX: Drop 'image/svg+xml' from allowedMimeTypes. OR strip with
          a server-side SVG sanitizer (sanitize-svg / DOMPurify-node).

  BUG-30  Asset uploads have no static route.
          AssetManager.getAssetUrl() returns /custom/${filename} but
          index.js never registers app.use('/custom', express.static(uploadDir)).
          Every uploaded asset is unreachable.
          FIX: Add the static route in setupRoutes(). Limit to image MIMEs
          via res.type check.

  BUG-31  multer 1.x has known CVEs.
          package.json     "multer": "^1.4.5-lts.1"
          FIX: Upgrade to multer v2 (npm install multer@^2). Test upload
          flow for breaking changes.

  Recommended approach: Build phases sequentially, finishing each phase completely

  Why sequential (not skeleton-first):
    1. Database schema is finalized in this plan — all tables, fields, indexes
       are documented upfront. No unknown references.
    2. Single developer — skeleton-first benefits teams working in parallel.
       For one person, incremental gives faster feedback.
    3. Phases are sequential by nature — Phase 0c creates DB, Phase 1 uses it,
       Phase 6+ builds on Phase 1 output. Clear dependency chain.
    4. Testing as you go — After Phase 0c, test DB creation. After Phase 1,
       test mining + balance logic. Early discovery of incompatibilities.
    5. Working code sooner — After Phase 0c (2h), you have a functional DB +
       wallet layer. After Phase 1 (8-10h), you have a working mining pool.

  Build order (strictly sequential):
    Phase 0c: Environment validation (DB schema, wallet wrappers, config)

    Phase 6:  Payment monitor (withdrawal queueing and retry logic)
              → Test: create withdrawal, verify DB state transitions
    Phase 8:  Reward distribution (scheduler, periodic payouts)
              → Test: scheduled payout runs, balances updated correctly

  ── Security problem in current layout ───────────────────────────────────

  Current: public_html/admin/  is served as static files by nginx.
  Anyone who knows the URL (/admin/users.html, /admin/miners.html) gets the
  full admin HTML — auth only fires when JS makes API calls. The DATA is
  protected by JWT (API returns 403 for non-admins), but the admin UI
  structure, endpoint names, and page layout are exposed to any visitor.
  This is "security through obscurity" — not acceptable.

  ── What changes ─────────────────────────────────────────────────────────

  1. pool-manager/ → server/
     Reason: clearer name; matches the pattern used in 052/053/054.

  2. admin HTML moves: public_html/admin/ → server/admin-panel/
     Reason: Express serves /admin/* with server-side requireAdmin check.
     Unauthenticated visitors get redirect to /login.html (not 403 JSON).
     Admin pages are never in the nginx web root.

  3. public_html gets app/ subdirectory for user-gated pages.
     These stay in nginx (data is JWT-protected at API level) but are
     logically grouped separately from fully-public pages.

  4. preview/ directory added for design mockups — never deployed.

  ── Auth cookie for admin page serving ───────────────────────────────────

  Current api.js stores JWT only in localStorage — not readable server-side.
  To allow Express to protect admin HTML, the login endpoint must also set:
    Set-Cookie: pool_session=<access_token>; HttpOnly; Secure; SameSite=Strict
  Express admin middleware reads the cookie (or Authorization header).
  api.js continues using localStorage for API calls — no behaviour change.

  Login flow after change:
    POST /api/auth/login
      → JSON response: { access_token, refresh_token }   ← existing
      → Set-Cookie: pool_session=<access_token>; HttpOnly ← new
    POST /api/auth/logout
      → Clear-Cookie: pool_session                        ← new
  Cookie is never read by JS (HttpOnly) — CSRF-safe because:
    /admin/* is GET-only (HTML pages) — no state change on the GET itself.
    All admin write actions go through /api/admin/* which require Bearer header.

  Express admin route:
    app.use('/admin', requireAdminPage, express.static(adminPanelDir));
    // requireAdminPage: reads pool_session cookie → verify JWT → check is_admin
    // if not authed: redirect /login.html   if not admin: redirect /

  ── Admin API Security (CRITICAL — based on grin-pool audit) ──────────────

  Admin API endpoints MUST enforce:
    1. HTTPS only (no HTTP) — TLS 1.2+ with strong ciphers
    2. Client certificate validation (mutual TLS) OR
       Session tokens stored server-side (NOT JWT — can't check revocation)
    3. Admin actions rate-limited: 5 reqs/min per IP + per session
    4. Every admin API call logged: timestamp, admin_id, endpoint, params (no secrets),
       response status, IP, User-Agent → admin_audit_log table
    5. Password change invalidates all active sessions (server-side token revocation)
    6. No hardcoded API keys in config.json (use environment variables only)

  Why not JWT for admin: JWT cannot be revoked mid-session. If an admin's
  token leaks, it remains valid until expiry (usually 24h). Server-side
  tokens can be invalidated immediately. Use stateless JWT for miners (public),
  stateful sessions for admins (security-sensitive).

  ── What is the stratum server? ──────────────────────────────────────────

  The Grin mining pool requires a stratum server to accept miner connections,
  parse share submissions, and notify the backend of blocks found.

  Current assumption (to be verified/clarified):
    · Use grin-mining-pool (C/C++, github.com/mimblewimble/grin-mining-pool)
    · OR write stratum server in Node.js (simpler, same language as backend)
    · Deployed as separate systemd service on the same VPS

  ── StratumMonitor client connection ──────────────────────────────────────

  Phase 6 monitor.js connects to stratum_host:stratum_port as a TCP client.
  It must:
    · Support persistent connection OR polling (clarify which)
    · Send/receive JSON-RPC 2.0 stratum messages
    · Receive share submissions: { "method": "submit", "params": [...] }
    · Receive block notifications: when pool finds a valid block
    · Parse miner username from initial handshake: "grin1abc...xyz.worker1"

  ── Stratum message format specification ────────────────────────────────

  Based on Grin stratum protocol (must verify against real server):

  Miner login:
    { "jsonrpc": "2.0", "id": 1, "method": "login",
      "params": { "login": "grin1abc...xyz.worker1", "pass": "x" } }

  Share submission:
    { "jsonrpc": "2.0", "id": 2, "method": "submit",
      "params": { "nonce": "...", "pow": [...], "height": <block_height> } }

  Block found notification (from server to monitor):
    { "jsonrpc": "2.0", "method": "job_notify",
      "params": { "height": <height>, "block_hash": "...", "status": "found" } }

  ── Stratum server deployment in pool setup ──────────────────────────────

  Phase 0c pool_configure() must:
    □ Prompt for stratum server type (grin-mining-pool vs custom)
    □ If using grin-mining-pool: verify binary exists (which grin-mining-pool)
    □ If custom: verify stratum_host:stratum_port is reachable (nc -zv test)
    □ Add stratum server as systemd service (if not already running)
    □ Configure stratum server to route shares to pool backend API endpoint
      (TBD: design REST endpoint for incoming shares from stratum server)

  ── Security assumption ──────────────────────────────────────────────────

  Stratum server is TRUSTED. It runs on the same VPS or private network.
  No authentication required between monitor and stratum server (assumes
  network isolation). If stratum server is remote, add IP whitelist + auth.

  miners/balances/withdrawals: keyed by grin_address, not user_id.

  ── Page categories ──────────────────────────────────────────────────────

  1. Public pages (no auth): / (home), /blocks, /payments, /miners, /faq, /help
     · Static or cached data (pool stats, block list) — OK to server-render
     · OR: Client-side fetch on mount (immediate page load, data populates ~1-2s)

  2. Miner pages (no auth): /account/:addr
     · Real-time hashrate + balance — needs live refresh
     · Needs polling or WebSocket for live updates
     · Client-side fetch recommended (useEffect)

  3. Admin pages (requireAdmin): /admin, /admin/users, /admin/miners, /admin/payment-stats
     · JWT token in localStorage — can't be read server-side
     · MUST be client-side fetch in useEffect + redirect if not authed
     · OR: use API route as proxy (server calls API with JWT from request context)
     · Recommended: client-side fetch (simple, matches existing api.js pattern)

  ── Recommendation ────────────────────────────────────────────────────────

  Use CLIENT-SIDE FETCH (useEffect + fetch API) for all pages:

    Pros:
      · Simpler code (no getServerSideProps complexity)

    export default function PageName() {

      const [loading, setLoading] = useState(true);
      const [error, setError] = useState(null);

      useEffect(() => {

          .then(r => r.json())
          .then(d => setData(d))
          .catch(e => setError(e))
          .finally(() => setLoading(false));

      if (error) return <ErrorCard />;
      return <Content data={data} />;

  NOTES & DECISIONS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  · Use better-sqlite3 (synchronous) not the async sqlite3 package.
    Reason: eliminates async complexity in db.js; matches 052/053 pattern;
    better-sqlite3 transactions are simpler and safer for financial data.

  · Drop SQLAlchemy ORM entirely — write raw SQL with better-sqlite3.
    Schema is simple and stable; no migration framework needed.

  · Keep the same SQLite schema (column names, types) so existing pool.db
    files are forward-compatible. New columns use ALTER TABLE ADD COLUMN
    with DEFAULT NULL — safe on existing databases.

  · node-fetch vs built-in fetch: Node 18+ has built-in fetch. Since the
    target is a Linux VPS, use built-in fetch (no extra dep). The version
    guard in pool_install() enforces Node ≥ 18.

  · The psutil import in admin_health is best-effort (optional).
    Node.js equivalent: use os.loadavg(), os.freemem(), os.totalmem().
    No npm dep needed — built-in os module covers disk + memory.

  · public_html/ (frontend HTML/CSS/JS) is NOT changed in the backend
    refactor phases (1–11). The API contract stays identical.
    Exception: new /api/pool/stats/regions endpoint is additive.
    Phase 13 (theme) touches CSS and button HTML only — no JS logic.

  · Share weight kept at 1.0 for now — no C32 difficulty weighting.
    Reason: the grin stratum protocol does not expose per-share difficulty
    in log lines. Would require parsing the stratum TCP stream directly.
    Mark as a future improvement if needed.

  · confirm_depth default = 10 blocks (~10 minutes at 60s block time).
    Grin has fast finality — orphan rate is low but not zero.
    Admin can set to 0 to restore immediate distribution behavior.

  · SQLite capacity estimate (for this pool):
    SQLite WAL mode handles 200–1 000 writes/second on a typical SSD VPS.
    Realistic Grin pool write load:
      — 50 active workers → ~8 writes/s (shares + heartbeats)
      — 500 active workers → ~80 writes/s  (well within limit)
      — 2 000+ active workers → write contention starts to appear
    Registered users (inactive) are unlimited — just rows in a table.
    Real ceiling: multi-region with separate stratum servers needing a
    shared DB across machines — SQLite is a single file, can't be shared
    over the network. At that point migrate to Postgres.
    Decision: start with SQLite, migrate to Postgres when/if needed.
    The schema is small and clean — migration is straightforward.

  ── Implementation notes (Claude) ────────────────────────────────────────

  Key decisions made when choosing this stack — record here so they don't
  get re-debated during implementation:

  1. App Router (not Pages Router)
     Use the Next.js 14+ App Router (app/ directory) not the older pages/.
     API routes live at app/api/.../route.ts. Server Components for pages
     that don't need interactivity, Client Components for forms and live
     data (dashboard, miners table, withdrawal flow).

  2. better-sqlite3 in API routes — important constraint
     Next.js must be self-hosted with `next start`, NOT deployed to Vercel
     or any serverless platform. better-sqlite3 is a native Node addon that
     requires a persistent process and a writable filesystem. The systemd
     service runs `next start -p 3002` as a long-lived process. This is
     identical to how Express ran before — just a different entry point.

  3. Singleton modules (monitor, scheduler) in Next.js
     monitor.js and scheduler.js cannot be imported directly into API route
     handlers — each request would create a new instance. Solution: use a
     module-level singleton initialized once via a startup hook.
     Next.js 14 has `instrumentation.ts` (register() hook) that runs once
     at server startup — use this to start monitor and scheduler.
     File: app/instrumentation.ts  →  import and start both singletons.

     Client Components that need auth state: read a NON-httpOnly `pool_user`
     cookie containing only {username, is_admin} — no secret, display only.
     This replaces the localStorage approach cleanly.

  7. Component library: shadcn/ui optional, used selectively

     but don't wrap everything — for simple elements write Tailwind directly.

  8. Data fetching pattern
     Public pages (home, recent blocks): Server Components, fetch at render.
     User pages (dashboard, miners): Client Components with SWR or
     React Query for polling (replaces the setInterval in the old JS).
     Admin pages: Client Components, polling every 15–30s.
     SWR is the lighter choice — one dep, works well for this pattern.
     Add: swr to package.json.

  9. Process manager: systemd (not pm2)
     Decision: keep systemd — consistent with every other Grin service in this
     toolkit (grin-node, grin-wallet, grin-drop, etc.). pm2 is an extra global
     install and adds operational complexity (pm2 startup, pm2-logrotate, etc.).
     Journalctl already handles log rotation for systemd units.

     Build-during-update downtime is acceptable: a mining pool update is a
     planned maintenance event. Miners reconnect automatically after restart.
     Expected downtime: time for `npm run build` (30–60s) + systemctl restart (~5s).

     Systemd unit template (write in pool_install()):
       [Unit]
       Description=Grin Pool Manager ($NETWORK)
       After=network.target

       [Service]
       Type=simple
       User=root
       WorkingDirectory=$POOL_APP_DIR
       ExecStart=/usr/bin/node_modules/.bin/next start -p $POOL_PORT
       Environment=NODE_ENV=production
       Environment=HOSTNAME=127.0.0.1
       Environment=GRIN_POOL_CONF=$POOL_CONF
       Restart=on-failure
       RestartSec=5s
       LimitNOFILE=65535

       [Install]
       WantedBy=multi-user.target

  10. When to migrate from SQLite to Postgres
      Trigger: need multiple stratum servers (different VPS regions) writing
      shares to a shared DB, OR > 2 000 concurrent active miners.
      Migration path: add Prisma ORM (schema.prisma mirrors current schema),
      run `prisma migrate` against Postgres, swap DATABASE_URL env var.
      The lib/db.js abstraction layer makes the swap mostly transparent
      to API routes — only db.js changes, not the route handlers.

  PAYMENT SYSTEM — Design, Security & Flowcharts
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  ── Balance model ────────────────────────────────────────────────────────────

  Two columns on miner_accounts, keyed by grin_address, always in sync:

    balance        Spendable GRIN — what the miner can withdraw right now
    balance_locked GRIN currently held in a pending withdrawal
                   (tor_checking, tor_sending, retry_scheduled)

  Invariant (enforced after every operation, checked in admin stats):
    SUM(miner_accounts.balance + miner_accounts.balance_locked)
    = SUM(shares.earned_grin) - SUM(confirmed_withdrawals.amount + fee)

  On withdrawal CREATE (Tor auto-pay):
    balance        -= amount + fee   (atomic, inside exclusive transaction)
    balance_locked += amount + fee

  On withdrawal CONFIRM:
    balance_locked -= amount + fee   (amount already gone from balance)

  On withdrawal TOR-FAIL (permanent, after max retries exhausted):
    balance        += amount + fee   (full reversal — miner never pays for failed tx)
    balance_locked -= amount + fee

  On withdrawal CANCEL (admin action):
    balance        += amount + fee
    balance_locked -= amount + fee

  All transitions log a row to balance_log (see schema additions).

  ── Minimum withdrawal ───────────────────────────────────────────────────────

  Default minimum: 10 GRIN — stored in cfg.min_withdrawal (pool.json).
  Single enforcement point: the API withdraw endpoint checks
    amount ≥ cfg.min_withdrawal → 400 if not.
  cfg.min_withdrawal IS the enforcement. No separate hard-coded value in code.

  pool_configure() defaults to 10 and validates operator input ≥ 10 before
  writing — this is operator UX guidance, not a security layer.

  Startup guard (config.js load()):
    if (!cfg.min_withdrawal || cfg.min_withdrawal <= 0)
      throw new Error('min_withdrawal missing or invalid in pool.json — fix config and restart')
  Server refuses to start with a broken config rather than silently accepting
  zero-GRIN withdrawals.

  The withdrawal fee (pool processing fee, separate from mining fee) is
  deducted on top of the amount:
    total_deducted = amount + cfg.withdrawal_fee
    user receives:  amount  (withdrawal_fee goes to pool_fee_address)

  ── Max 1 pending withdrawal per address ────────────────────────────────────

  At CREATE time, check:
    SELECT COUNT(*) FROM withdrawals
    WHERE grin_address = ? AND status IN
    ('tor_checking','tor_sending','retry_scheduled') > 0
    → 429 "You have a pending withdrawal in progress."
  Reason: prevents balance-lock abuse (miner can't lock all their GRIN in
  limbo repeatedly) and simplifies state tracking.

  ── Griefing vector: public endpoint ──────────────────────────────────────

  POST /api/account/:addr/withdraw is completely public (no auth required,
  address-as-identity model). An attacker knowing a miner's address can
  trigger a premature withdrawal, locking the miner out of new withdrawals
  until the pending one resolves (~7 days max).

  TRADEOFF ACCEPTED: Funds always go to the miner's own Grin Tor address.
  An attacker cannot redirect or steal. Griefing is annoying (locked for days)
  but not financially harmful. The 1-pending limit and retry mechanism
  eventually release the lock.

  Alternative (not implemented): add per-address cooldown on withdrawal
  creation (e.g., skip if last withdrawal created < 24h ago), but this
  adds config complexity and slows legitimate miners.

  PAYMENT METHOD — PHASE 1 vs PHASE 2
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  PHASE 1 (this refactor — ship first, run stable miners on it):
    Tor automatic payment only.
    Pool initiates all payouts. Miner wallet just needs to be listening.
    Simple, production-grade, no interactive steps required.

  PHASE 2 (future, after pool is live and stable):
    Slatepack interactive withdrawal added as an opt-in option.
    Requires email-verified account registration.
    Not part of this refactor — deferred to keep Phase 1 clean and shippable.

  While Slatepack is not yet implemented on this pool, miners who want to
  use Slatepack withdrawal today can refer to:

    🔗 https://gaeapool.com/#/

  GaeaPool supports Slatepack withdrawal and serves as a good reference for
  how the interactive transaction flow works in practice.
  Slatepack will be added to this pool in a future update.

  PAYMENT FLOW — Tor Automatic with Retry Queue  [PHASE 1]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Miners never log in. Their Grin address IS their identity. Withdrawals are
  triggered automatically when balance ≥ cfg.auto_payout_threshold (default 10
  GRIN), OR manually via the public account page at:
    pool.domain/account/<grin_address>

  There is no Slatepack flow. Tor auto-pay is the only payment method.
  If the miner wallet is offline, the pool retries on a backoff schedule
  before permanently failing and returning funds.

  ── TRIGGER: Auto-payout (scheduler, every 6h) ──────────────────────────

  SELECT * FROM miner_accounts
  WHERE balance >= (cfg.auto_payout_threshold + cfg.withdrawal_fee)
  AND NOT EXISTS (
    SELECT 1 FROM withdrawals
    WHERE grin_address = miner_accounts.grin_address
    AND status IN ('tor_checking','tor_sending','retry_scheduled')
  )
  For each row → initiate withdrawal (same flow below).
  NOTE: include withdrawal_fee in threshold check to avoid creating withdrawals
        that will fail the balance check at send time.

  ── TRIGGER: Manual withdraw (public account page) ───────────────────────

  POST /api/account/:addr/withdraw  { amount }   (no auth — address is identity)

  VALIDATION (before any DB write):
    · :addr must be a valid Grin address format       → 400 if not
    · amount ≥ cfg.min_withdrawal                     → 400 if not
    · amount + fee ≤ miner_accounts.balance           → 400 if not
    · no pending withdrawal for this grin_address     → 429 if exists

  ── RETRY SCHEDULER (node-cron, every 30 minutes) ────────────────────────

  SELECT * FROM withdrawals
  WHERE status = 'retry_scheduled'
  AND next_retry_at <= unixepoch()

  For each row: re-enter the flow at the "wallet.checkWalletOnline()" step.

  ── PERMANENT FAILURE (retry_count ≥ 4, ~7 days patience) ───────────────

  ┌───────────────────────────────────────────────────────────────────┐
  │  DB EXCLUSIVE TRANSACTION                                         │
  │    UPDATE miner_accounts SET                                      │
  │      balance        = balance + (amount + fee),  ← full reversal  │
  │      balance_locked = balance_locked - (amount + fee)             │
  │    UPDATE withdrawals SET status='tor_failed'                     │
  │    INSERT balance_log (type='reversal', reference=withdrawal.id)  │
  │    INSERT withdrawal_events (to_status='tor_failed',              │
  │      note='max retries exhausted after 7 days')                   │
  └───────────────────────────────────────────────────────────────────┘

  Miner's balance is fully restored. No fee charged for failed payout.
  Admin sees tor_failed withdrawals on payment-stats page.
  Admin can manually trigger a retry via POST /api/admin/withdrawals/:id/retry.

  ── PUBLIC ACCOUNT PAGE ──────────────────────────────────────────────────

  URL:  pool.domain/account/<grin_address>
  Auth: none — public stats (like 2miners.com/account/<addr>)

  Displays:
    · Current hashrate (live, 1h average, 24h average)
    · Balance available + balance locked in pending withdrawal
    · Active workers (worker_name, last_seen, hashrate)
    · Recent blocks found (time, reward)
    · Payout history (last 20 confirmed withdrawals with tx date + amount)
    · Pending withdrawal status (if any): status, retry_count, next_retry_at
    · Wallet status probe: "Check wallet" button → GET /api/account/:addr/tor-check

  Note: full reversal means miner pays NO fee for a failed withdrawal.
  Fee is only deducted on confirmed payouts.

                ┌───────────────────────────────────────┐

                │ (balance reversal on cancel)          │
                └───────────────────────────────────────┘

             Cannot cancel a tor_sending or confirmed withdrawal.

    0 →  6h
    1 → 12h
    2 → 24h
    3 → 48h
    4 → permanent fail (tor_failed)

  Sensitive admin actions (withdrawal retry/cancel) require re-authentication
  to prevent cross-site request forgery and session hijacking.

  ── Design: JWT age check (simple, no extra tokens) ────────────────────

  POST /api/admin/withdrawals/:id/retry  (requires sensitive re-auth)
  POST /api/admin/withdrawals/:id/cancel (requires sensitive re-auth)

  Implementation:
    1. Client sends request with Authorization header (existing JWT)
    2. Server checks JWT age: iat = token issue time
    3. If age < 5 minutes: allow (user actively logged in, likely same session)
    4. If age ≥ 5 minutes: require fresh password verification
       Return 403: { error: "reauthenticate", challenge_required: true }
       Client shows password prompt modal
       User submits password → POST /api/auth/reauth { password }
       Server returns: { reauth_token: <jwt_with_iat_reset> } (valid for 5 min)
       Client retries original request with new reauth_token in header

  Alternative (if above is too complex):
    · Just check JWT age, no password prompt
    · Sessions older than 5 min get 403 "please log in again"
    · User navigates to /admin, logs in fresh, retries action
    · Simpler UX but requires page reload

  ── Recommendation ────────────────────────────────────────────────────

  Use the JWT age check approach (5 min threshold):
    · No additional password storage needed
    · No state in server (stateless JWT)
    · Clear UX: "Your session expired, please log in again"
    · Compatible with existing auth.js

  ── Phase 4 auth.js addition ──────────────────────────────────────────

  Add middleware: requireFreshAuth(maxAgeSeconds = 300)
    const iat = token.iat  (issued at time, in JWT payload)
    const age = Math.floor(Date.now() / 1000) - iat
    if (age > maxAgeSeconds) return 403 { challenge_required: true }
    else return next()  (allow)

  ── Phase 10 API routes ──────────────────────────────────────────────

  POST /api/admin/withdrawals/:id/retry
    middleware: [requireAdmin, requireFreshAuth(300)]
    if 403 challenge_required: client shows "log in again" modal
    if 200: proceed with retry

  POST /api/admin/withdrawals/:id/cancel
    middleware: [requireAdmin, requireFreshAuth(300)]
    same flow

  - github.com/mining-pool/open-grin-pool (simplified monolithic pool)
  - github.com/grin-pool/grin-pool-monitor (stratum monitoring reference)

  Key validations & corrections applied from real implementations:

  ✅ CONFIRMED CORRECT:
    · Address-as-identity model works (grin-pool proves this)
    · Slate-based Tor delivery is viable (existing implementation)
    · Per-miner balance tracking + fee deduction is standard
    · PPLNS reward distribution is battle-tested

  ⚠️ CRITICAL CORRECTIONS:
    1. Confirm-depth: Changed from 10 blocks → 1441 blocks (grin-pool standard)
       Why: 10 blocks (~40 min) insufficient to prevent reorg payouts
    2. Port probing removed: Skipping checkWalletOnline() TCP probe
       Why: Leaks Tor circuit identity (grin-pool has this bug)
    3. Orphan detection added: New orphanDetection() job
       Why: Validates nonce against chain after payout (prevents orphan steals)
    4. Admin API security: Requires HTTPS + session tokens (not JWT)
       Why: grin-pool audit shows JWT tokens can't be revoked mid-session
    5. Retry backoff: Already correct (age-based, not amount-based)
       grin-pool has fairness issue we avoided

  ⚠️ ARCHITECTURAL NOTES:
    · grin-pool uses 1441 blocks (~1 day) maturity, but allows config override
      We default to 1441 for safety, operator can lower if they understand reorg risk
    · Both pools support Slatepack (slate files) as alternative to HTTP
      Our Tor-only approach is simpler but requires miner to run listener/re-fetch
    · No existing pool explicitly supports Tor-only without manual intervention
      This is more privacy-focused but adds UX friction vs HTTP pools

    □ Audit all balance_log entries for reconciliation correctness

  CRITICAL PRE-LAUNCH CHECKLIST (NON-NEGOTIABLE)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  These items MUST be completed and tested before ANY deployment (testnet or mainnet).
  Failure to complete any of these introduces unacceptable risk of:
    · Payouts to orphaned blocks (loss of pool capital)
    · Race conditions causing double-payouts or missing credits
    · Wallet transaction failures without recovery path
    · Miner balance ledger inconsistencies leading to disputes

  ── 1. BLOCK MATURITY & ORPHAN DETECTION ────────────────────────────────────

  [REQUIRED]
    ☐ Add block.nonce field to blocks table (INTEGER NOT NULL)
    ☐ Query Grin node API get_header to fetch nonce during block save
    ☐ Implement confirmPendingBlocks() to:
        · Query node.get_outputs() for each block nonce
        · If nonce not found in chain (UTXO spent): mark block as orphaned
        · Reverse all payouts for orphaned block (balance_log reversal events)
        · No manual operator action — automatic recovery
    ☐ Schedule orphan detection job to run every 6 hours (cron 0 */6 * * *)
    ☐ Job queries blocks where status='immature' AND height <= (tip - 1441)
        and performs nonce validation above

  [TESTING]
    ☐ Testnet reorg scenario:
        · Mine a block on pool, confirm it appears in blocks table
        · Simulate reorg: start an alternate chain tip (use two nodes if possible)
        · Verify orphan detection job marks block as orphaned within 6h
        · Verify all payouts for that block are reversed in balance_log
        · Verify total_paid_out accounting is corrected
    ☐ No blocks should reach 'paid' status with nonce NOT present in chain

  ── 2. BALANCE LOCKING & RACE CONDITIONS ────────────────────────────────────

  [REQUIRED]
    ☐ Update User.balance and User.balance_locked UPDATEs to use
        .with_for_update() (SELECT FOR UPDATE in raw SQL)
    ☐ All balance mutations must be wrapped in transaction:
        BEGIN;
        SELECT balance, balance_locked FROM miner_accounts WHERE addr=? FOR UPDATE;
        -- verify amount is available
        UPDATE miner_accounts SET balance = balance - amount, ...
        INSERT INTO balance_log ...
        COMMIT;
    ☐ Test with concurrent clients:
        · Write load test that submits 10 withdrawals simultaneously
        · Verify final balance is correct (no double-credits or missing debits)
        · Verify balance_log ledger sums to expected balance

  [TESTING]
    ☐ Concurrent withdrawal load test (100 parallel requests, same miner)
    ☐ Verify final balance = initial - total withdrawn (within floating-point epsilon)
    ☐ Verify each withdrawal decremented balance atomically

  ── 3. MINER AUTO-CREATION RACE CONDITION ────────────────────────────────────

  [REQUIRED]
    ☐ Stratum handler (newWorker event) uses atomic insert:
        INSERT OR IGNORE INTO miner_accounts (grin_address, ...)
        VALUES (?, 0.0, 0.0, ...)
        on connection setup (Phase 0c, stratum.js)
    ☐ If miner already exists, INSERT OR IGNORE does nothing (safe)
    ☐ Do NOT check-before-insert (race window)
    ☐ Test scenario:
        · Two stratum connections arrive for same grin_address simultaneously
        · Verify only one row created, second connection reuses existing row
        · Verify no transaction errors

  [TESTING]
    ☐ Concurrent login load test (same miner address, 5 simultaneous connections)
    ☐ Verify exactly one row in miner_accounts for that address
    ☐ Verify no PRIMARY KEY UNIQUE constraint violations

  ── 4. WALLET INTEGRATION & IDEMPOTENCY ────────────────────────────────────

  [REQUIRED]
    ☐ Identify whether grin-wallet send() is idempotent:
        Test: Send tx to address A, let it fail partway through
               Restart wallet (or app), attempt send again
               → Does wallet detect it already created the tx?
               → Or does it create a duplicate?
    ☐ If idempotent: Document the behavior (which endpoint detects duplicates)
    ☐ If NOT idempotent:
        · Implement transaction ID caching (txid per withdrawal_id)
        · Before send: check if this withdrawal already has a txid in DB
        · On retry: reuse same txid (verify with wallet.get_tx(txid))
        · Never allow two separate txids for same withdrawal

  [TESTING]
    ☐ Testnet Tor send with intentional failure (kill wallet mid-send)
    ☐ Restart pool app, trigger retry
    ☐ Verify final result is single transaction, not duplicates
    ☐ Check blockchain for duplicate outputs (should be none)

  ── 5. STRATUM WORKER NAME VALIDATION ────────────────────────────────────────

  [REQUIRED]
    ☐ Add regex validation for stratum username parameter:
        Allowed: [a-z0-9]{5,} (at minimum 5 alphanumeric, lowercase)
        Reject any special chars, spaces, etc.
    ☐ Miner login format enforced: grin_address.worker_name or grin_address
    ☐ validate on stratum handler:
        if (!/^grin1[a-z0-9]+(\.[a-z0-9]+)?$/.test(username)) {
          close stratum connection with auth_error
        }
    ☐ Log rejected usernames in debug logs (not security-critical, but helpful)

  [TESTING]

    ☐ All invalid attempts should disconnect cleanly

  ── 6. NODE API FAILURE HANDLING ────────────────────────────────────────────

  [REQUIRED]
    ☐ Document failure modes:
        1. Node API timeout (> 5s) — log warning, retry after 30s
        2. Node API returns error (e.g., "Method not found") — log error, escalate
        3. Connection refused (node down) — circuit breaker pattern
           After 3 consecutive failures: exponential backoff (30s, 60s, 120s, 300s)
           Once successful: reset counter to 0
    ☐ Do NOT kill pool on transient node failures
    ☐ Queue up work (block confirmations, payouts) and retry when node is back
    ☐ Health check endpoint: GET /api/admin/health returns node status

  [TESTING]
    ☐ Stop Grin node, verify pool stays running (queues work)
    ☐ Restart Grin node, verify pool catches up (no blocks lost)
    ☐ Trigger node timeout artificially (firewall rule), verify retry logic

  ── 7. WALLET STDOUT PARSING FRAGILITY ──────────────────────────────────────

  [REQUIRED] (Phase 1, but document the risk)

    ☐ Problem: grin-wallet version differences may change output format
    ☐ Action item: Before Phase 1 deployment, verify this format with:

    ☐ Future migration: Switch to JSON-RPC API (if available) instead of stdout
    ☐ Fallback: If parsing fails, log full stdout and manual_review flag in DB

  [TESTING]

    ☐ Intentionally pass malformed JSON, verify error handling

  ── 8. CONFIRM-DEPTH CONFIGURATION ─────────────────────────────────────────

  [REQUIRED]
    ☐ Default: confirm_depth = 1441 blocks (mainnet safety)
    ☐ Testnet: confirm_depth = 100 blocks (faster feedback during testing)

        "current_network": "testnet",  ← operator must set this
    ☐ Operator must understand reorg risk before lowering confirm_depth

  [TESTING]

    ☐ Testnet: block ready for payout after 100 confirms (~6 hours)
    ☐ Mainnet: block ready for payout after 1441 confirms (~5 days)

  ── 9. BALANCE LEDGER RECONCILIATION ────────────────────────────────────────

  [REQUIRED]
    ☐ Run admin reconciliation query daily (cron 0 3 * * * — 3:00 UTC):
        SELECT grin_address,
               SUM(amount) FILTER (WHERE event_type='credit') as total_credit,
               SUM(amount) FILTER (WHERE event_type='debit') as total_debit,
               SUM(amount) FILTER (WHERE event_type='reversal') as total_reversal,
               balance as current_balance
        FROM balance_log bl
        JOIN miner_accounts ma ON bl.grin_address = ma.grin_address
        GROUP BY grin_address
        HAVING (total_credit - total_debit - total_reversal) != current_balance
    ☐ If any row fails check: send alert to admin
    ☐ Do NOT auto-fix — operator must investigate and approve correction

  [TESTING]

    ☐ Introduce intentional mismatch (corrupt balance manually)
    ☐ Verify reconciliation query detects it

  [REQUIRED]
    ☐ Admin login requires HTTPS (no plaintext password over HTTP)
    ☐ Session tokens: JWT with 1-hour expiry (not stateless after logout)
    ☐ Sensitive operations (withdrawal retry/cancel) require fresh auth:

        · If JWT age ≥ 5 min: challenge with password re-entry
    ☐ All admin actions logged to admin_audit_log (with IP, timestamp, actor_id)
    ☐ No admin action should succeed without audit log entry
